# Darklock Guard IPC Security

- **Transport**
  - Linux: Unix domain socket `/run/darklock/ipc.sock` (mode 0600, owner = service user).
  - Windows: Named pipe `\\.\pipe\DarklockGuard` with ACL restricted to Administrators + SYSTEM.
  - Optional TCP fallback: bind `127.0.0.1` *only* after master key is loaded; still requires session token.

- **Master secret**
  - Generated by the service on first start and stored in OS keyring; never exposed to the GUI.

- **Session flow**
  1. GUI calls Tauri command `request_session`.
  2. Backend sends a nonce to the service over the protected socket/pipe.
  3. Service derives `session_token = HMAC(master_key, nonce || timestamp || device_id)` with a short TTL (5–10 minutes).
  4. GUI stores `session_token` in memory only and presents it on every IPC call.
  5. Service validates token + timestamp and tracks `request_id` values per token to prevent replay within TTL (LRU set).
  6. GUI restart ⇒ previous tokens expire at TTL; a new session must be requested.

- **Replay & stale protection**
  - Each request must include `{session_token, request_id, timestamp}`.
  - Reject if timestamp skewed, token expired, or `request_id` already seen within TTL.

- **Website relay guard**
  - Remote requests must include `device_id`, `request_id`, `timestamp`.
  - Allowlist: status fetch, scan request, verify chain.
  - Service blocks all remote requests when GUI is absent or declines the approval prompt.

- **First-run/Trust**
  - Service refuses website relay and won’t emit “Protected” until onboarding is complete in backend state.
